—
Where the numbers come from: outcome classification uses the canonical Knowify rules (applyKnowifyRules) — same as every other page. Rejected→loss, Active→win, Closed with $0 invoiced→loss (unbilled), Bidding past stale-threshold→loss, otherwise pending. Relationship-channel jobs are excluded from the rate. Live aliases merge same-GC name variants; excluded names (the company itself, vendors, location strings, project descriptors) are dropped entirely. GCs below the min-bids floor are filtered out.

Wins / losses by year—
▸Current pending bids · expected wins by GC forward-looking · pending count × historical WR · multi-GC count carved out
What this is: for each GC with an active pending bid, the historical WR (count + $) is applied to that GC's pending volume to estimate likely wins. Multi-GC bids are flagged separately — these are projects where multiple GCs were collecting bids on the same opportunity, so a "loss" through one of them isn't necessarily a loss against SFS — only one GC ultimately gets the project. Below, GCs with fewer than 5 historical decided bids show no rate (not enough sample).

Multi-GC pending = bids on projects where multiple GCs are also bidding on the same project (job name with the GC stripped is shared). Only one GC ultimately gets the project; SFS may bid through multiple of them. The expected-wins formula doesn't discount these (the historical WR already absorbs them implicitly), but the count is shown so you know how much of the pending volume has multi-GC ambiguity.
